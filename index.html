QJSON

JSON where 0.1 + 0.2 = 0.3

pip install qjson npm install qjson

The problem

JSON numbers are IEEE 754 doubles. 0.1 + 0.2 gives 0.30000000000000004. Financial data, sensor readings, scientific measurements — anything that needs decimal precision, JSON can't represent faithfully.

Three suffixes fix it

67432.50M            // BigDecimal -- exact base-10 decimal
9007199254740993N    // BigInt -- arbitrary-precision integer
3.14159265358979L    // BigFloat -- arbitrary-precision binary float

No new syntax to learn. Valid JSON is valid QJSON.

The trick: indexed precision

Every number — even one with 300 digits — gets stored as a three-column interval: [lo, str, hi].

  • lo = largest IEEE double ≤ the exact value
  • hi = smallest IEEE double ≥ the exact value
  • str = exact decimal string (NULL when lo == hi)
The SQL index works on the doubles. 99.999% of comparisons resolve from the index alone. The 0.001% that land in the overlap zone fall through to exact string comparison. Arbitrary precision at indexed speed.

Solve backwards

Store 3 of 4 values, leave one as ?. The solver fills it in — in any direction:

present = 10000M, rate = 0.05M, periods = 10, future = ?
  --> future = 16288.946267774414M

present = ?, rate = 0.05M, periods = 10, future = 16288.95M
  --> present = 10000.002290952...M

Works with SQRT, EXP, LOG, SIN, COS, TAN and their inverses. All arbitrary precision via libbf.

Datalog in JSON

// Facts as sets of tuples
{
  parent: {[alice, bob], [bob, carol], [carol, dave]}
}

// Grandparent join
.parent[K1][1] == .parent[K2][0]

// Transitive closure
qjson_closure(root, '.parent')
// --> a->b, a->c, a->d, b->c, b->d, c->d

[K] iterates both arrays and sets. Bare identifiers are strings. qjson_closure uses WITH RECURSIVE. Handles cycles.

What else

Encrypted storage

SQLCipher page-level encryption. WASM + OPFS for browser. One key, everything encrypted.

Complex keys

Any value as object key. {42: "answer"}, {[1,2]: "pair"}.

Binary blobs

0jSGVsbG8 — JS64 encoding, identifier-safe, no escaping needed.

One extension

Everything in a single SQLite extension. No server, no daemon, no JVM.
