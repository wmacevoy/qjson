2. Arbitrary-precision numerics

The problem: IEEE 754 doubles

JavaScript

Plain JSON numbers are IEEE 754 doubles. 0.1 + 0.2 gives 0.30000000000000004. That's not a bug — it's how binary floating-point works.

The fix: M and L suffixes

QJSON SQL — arbitrary-precision arithmetic

QJSON adds three arbitrary-precision numeric types. Plain numbers (3.14) remain IEEE 754 doubles.

42N                  // BigInt — arbitrary-precision integer
67432.50M            // BigDecimal — exact base-10 decimal
3.14159265358979L    // BigFloat — arbitrary-precision binary float

3. Unbound variables — pattern matching + solving

The ? prefix marks an unknown value:

QJSON input

An unbound variable matches anything in queries and can be solved by a constraint formula.

The solver

Compound interest: FV = PV × (1 + r)n

Solve — edit values, leave one as ?

Try changing which field is ? — the same formula solves in any direction:

  • Leave future = ? → computes future value
  • Leave present = ? → computes present value
  • Leave rate = ? → finds the rate (nth root)
  • Leave periods = ? → finds the time (logarithm)
The formula .future == .present * POWER(1 + .rate, .periods) is decomposed into 3-term constraints. The solver uses inverse operations (division for multiplication, logarithm for power) automatically. All arithmetic uses libbf arbitrary-precision — no IEEE 754 rounding when using M/L types and qjson_* functions.
